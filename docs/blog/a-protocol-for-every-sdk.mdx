An SDK has two jobs that look similar but are not the same. It gives application
code a pleasant typed function, and it speaks a particular service's language
over the network. The Distilled architecture draws a firm line between those
jobs using [Effect](https://effect.website/docs).

This article expands on the [`v0.6.0` release notes](/changelog/v0.6.0/), the
[Protocol architecture decision](/guides/protocol-architecture), and the
[runtime reference](/reference/runtime).

Here, `v0.6.0` names the coordinated site release. The packages published in
that release use the `0.2.x` version line.

Our architectural reference is
[Alchemy's Distilled repository](https://github.com/alchemy-run/distilled).
The established package line is visible on
[`main`](https://github.com/alchemy-run/distilled/tree/main), while the Protocol
architecture lives on
[`beta`](https://github.com/alchemy-run/distilled/tree/beta). Alchemy's exact
[`2f748184` beta commit](https://github.com/alchemy-run/distilled/commit/2f748184b7774370c6599bd7d3b94711e68581b0)
is the pinned architectural reference for this release.

## What changed

For the seven OpenAPI-backed SDKs, every generated operation now describes what
it needs: its input and output types, HTTP method and path, request fields,
possible errors, and Effect requirements. Our shared
[`API.make` operation model](https://github.com/kevinmichaelchen/distilled/blob/9fd0c08fab7885ee453af5634826d20e26024759/src/api.ts#L20-L255)
joins that description to a provider Protocol. The
[HTTP trait machinery](https://github.com/kevinmichaelchen/distilled/blob/9fd0c08fab7885ee453af5634826d20e26024759/src/protocol-http.ts#L271-L434)
handles common request structure, while each provider owns credentials, base
URLs, headers, response envelopes, and error messages. Alchemy's
[`API.make`](https://github.com/alchemy-run/distilled/blob/2f748184b7774370c6599bd7d3b94711e68581b0/packages/core/src/api.ts)
and
[HTTP trait implementation](https://github.com/alchemy-run/distilled/blob/2f748184b7774370c6599bd7d3b94711e68581b0/packages/core/src/protocol-http.ts)
are the upstream references for that split.

For those providers, the shared engine joins them at runtime:

```text
typed operation → provider Protocol → Effect HTTP client → provider Protocol → typed result
```

[GitHub](/sdks/github) and [Jira](/sdks/jira) can therefore share operation
semantics without pretending that GitHub's bearer token and Jira's basic
authentication are the same thing. [Slack](/sdks/slack) follows the same
resource-grouped service layout, but its
[Slack Protocol](https://github.com/kevinmichaelchen/distilled-slack/blob/8bdb5078d063aee953902d4ffa962898c800d3cf/src/protocol.ts)
deliberately bypasses the HTTP-trait path and delegates transport to Slack's
official typed
[`WebClient`](https://github.com/slackapi/node-slack-sdk/blob/70e15b8442745d215f9ae259a2c970b306aff51a/packages/web-api/src/WebClient.ts).
Alchemy's
[Cloudflare Protocol](https://github.com/alchemy-run/distilled/blob/2f748184b7774370c6599bd7d3b94711e68581b0/packages/cloudflare/src/protocol.ts)
shows the same provider-owned boundary in the upstream beta branch.

## Why this is better

For OpenAPI providers, the separation makes behavior easier to reason about.
Credentials are resolved for each call, so rotation and refresh can work
without reconstructing provider state. Errors say which provider and transport
failures can actually occur. Request routing comes from the
[schema-trait contract](/guides/protocol-architecture#decision) rather than a
generic body-building guess. Retry behavior is attached only to safe methods
unless a provider makes an explicit decision for a mutation. Slack gets the
same per-call credential resolution and typed Effect failures through its
WebClient adapter.

Generation is safer too. A broken source or patch cannot leave a half-updated
SDK behind while still reporting success. The
[OpenAPI generator](/reference/generator) builds a complete staging tree,
verifies every operation, and swaps it into place only after the whole run
succeeds.

## What this means for users

The `0.2.0` provider packages group operations by resource and make them
available from the root `Services` namespace or a direct service subpath. The
shared engine is published as
[`@kevinmichaelchen/distilled@0.2.1`](https://www.npmjs.com/package/@kevinmichaelchen/distilled),
and the [support matrix](/sdks/support-matrix) links every provider package.

### What changed in TypeScript

The change is visible in application code, not only inside the factory. In the
seven OpenAPI SDKs, flat operation exports moved into resource services. For
operations with a JSON request payload, path and query values stay at the top
level while the payload now sits under `body`.

<Diff
  lang="ts"
  old={`import { pullsCreate } from "@kevinmichaelchen/distilled-github";

const pullRequest = yield* pullsCreate({
  owner: "acme",
  repo: "payments",
  title: "Reduce checkout latency",
  head: "automation/checkout-cache",
  base: "main",
  body: "Adds a bounded cache around the pricing lookup.",
});`}
  new={`import { Services } from "@kevinmichaelchen/distilled-github";

const pullRequest = yield* Services.pulls.pullsCreate({
  owner: "acme",
  repo: "payments",
  body: {
    title: "Reduce checkout latency",
    head: "automation/checkout-cache",
    base: "main",
    body: "Adds a bounded cache around the pricing lookup.",
  },
});`}
/>

The same operation can be imported directly from the resource subpath—for
example, `@kevinmichaelchen/distilled-github/pulls`. OpenAPI operations remain
both directly callable and yieldable; that behavior did not begin with this
release.

The generated declaration now names the complete input, output, error, and
Effect requirements. This is the actual GitHub declaration shape, with only
imports and line wrapping elided:

<Diff
  lang="ts"
  old={`export declare const reposGet: OperationMethod<
  ReposGetInput,
  ReposGetOutput,
  Forbidden | NotFound,
  Credentials,
  never
>;`}
  new={`export type ReposGetError =
  | Forbidden
  | NotFound
  | GitHubOpError;

export declare const reposGet: API.OperationMethod<
  ReposGetInput,
  ReposGetOutput,
  ReposGetError,
  GitHubOpContext
>;`}
/>

`ReposGetError`, `GitHubOpError`, and `GitHubOpContext` are real provider
exports rather than explanatory aliases:

```ts
export type GitHubOpError =
  | DefaultErrors
  | ConfigError
  | HttpClientError.HttpClientError;

export type GitHubOpContext = Credentials | HttpClient.HttpClient;
```

The newer error union includes the declared endpoint failures plus provider,
configuration, parsing, and transport failures. The context tells Effect that
the operation needs credentials and an HTTP client. Missing environment
credentials now produce a typed `ConfigError` instead of an Effect defect. See
the generated [`0.1.x` operation](https://github.com/kevinmichaelchen/distilled-github/blob/eafa531af5710892e38ce31d457cfc1d24ffe785/src/operations/reposGet.ts),
the [`0.2.x` operation](https://github.com/kevinmichaelchen/distilled-github/blob/ceb717aeb89e2c32c16d089e007bfdfa8c4a7ab8/src/services/repos.ts#L2216-L2228),
and its [provider contract](https://github.com/kevinmichaelchen/distilled-github/blob/ceb717aeb89e2c32c16d089e007bfdfa8c4a7ab8/src/protocol.ts#L31-L36).
These operation signatures are more truthful, although the
[Protocol failure boundary](#what-still-deserves-work) still has room to become
fully compiler-derived.

</Expandable>

Slack follows the same resource-grouped layout, but keeps the official
`WebClient` input and output types. Its public layer changed from
`SlackClientLive` to `SlackProtocolLive`, and its operation errors now include
credential failures. The [runtime reference](/reference/runtime),
[GitHub SDK guide](/sdks/github), and [Slack SDK guide](/sdks/slack) cover the
current contracts and complete setup.

## Where the factory stands

The [eight SDKs](/sdks) now expose 4,066 operations from their latest pinned
sources. [OpenSearch](/sdks/opensearch) gained ten HEAD operations that its
contract had always described. All OpenAPI providers regenerate with zero
failed or unsupported operations. Publishing then typechecks, tests, and builds
the committed projection before it reaches npm. The implementation is
reviewable in the
[`kevinmichaelchen/distilled` repository](https://github.com/kevinmichaelchen/distilled);
the Protocol architecture entered the shared core in
[`2b37201`](https://github.com/kevinmichaelchen/distilled/commit/2b3720159878776fea9c74fa948e2da7db540ff5).

## What still deserves work

The architecture is stronger, but the work is not finished. We still want a
fully typed Protocol error boundary, richer nested-query encoding, clearer
schema-aware output decoding, synchronized OAuth refresh caching, and live
provider integration tests in addition to deterministic transport fixtures.
Alchemy's [Protocol work on `beta`](https://github.com/alchemy-run/distilled/tree/beta)
is still evolving, so future releases should continue to follow its public
operation shape while treating unreleased implementation details as
provisional. The
[unlisted beta audit](https://gist.github.com/kevinmichaelchen/67508b68a84402d3163413e3f3e826b5)
records the remaining upstream questions with exact source links.

Those are bounded improvements now. The important part is already structural:
one operation model, one shared engine, and a provider Protocol wherever APIs
genuinely differ.
